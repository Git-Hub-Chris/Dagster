# Dagster project file reference

Following [our recommended best practices for project structure](/guides/dagster/recommended-project-structure), a small Dagster project might look like this:

```shell
.
├── README.md
├── my_dagster_project
│   ├── __init__.py
│   ├── assets
│   │   ├── __init__.py
│   │   ├── activity_analytics
│   │   │   ├── __init__.py
│   │   │   └── activity_forecast.py
│   │   └──  core
│   │   │   ├── __init__.py
│   │   │   ├── id_range_for_time.py
│   │   │   └── items.py
│   ├── jobs.py
│   ├── partitions.py
│   ├── resources
│   │   ├── __init__.py
│   │   ├── hn_resource.py
│   │   └── snowflake_io_manager.py
│   ├── sensors
│   │   ├── __init__.py
│   │   ├── hn_tables_updated_sensor.py
│   │   └── slack_on_failure_sensor.py
│   └── utils
├── my_dagster_project_tests
├── pyproject.toml
├── setup.cfg
├── setup.py
└── tox.ini
```

## Example project structures

### Local development

When running Dagster locally, you can configure `pyproject.toml` to define a single code location. If you have multiple code locations, list them in a single `workspace.yaml` file. You can also include a `dagster.yaml` to configure things like storage, run launchers, and concurrency limits.

<TabGroup>
<TabItem name="Single code location">

#### Single code location

For local development, a project with a single code location might look like this:

```shell
.
├── README.md
├── my_dagster_project
│   ├── __init__.py
│   ├── assets
│   │   ├── __init__.py
│   │   ├── activity_analytics
│   │   │   ├── __init__.py
│   │   │   └── activity_forecast.py
│   │   └──  core
│   │   │   ├── __init__.py
│   │   │   ├── id_range_for_time.py
│   │   │   └── items.py
│   ├── jobs.py
│   ├── partitions.py
│   ├── resources
│   │   ├── __init__.py
│   │   ├── hn_resource.py
│   │   └── snowflake_io_manager.py
│   ├── sensors
│   │   ├── __init__.py
│   │   ├── hn_tables_updated_sensor.py
│   │   └── slack_on_failure_sensor.py
│   └── utils
├── my_dagster_project_tests
├── dagster.yaml 			## optional, used for instance settings
├── pyproject.toml			## defines a single code location
├── setup.cfg
├── setup.py
└── tox.ini
```

In this example project, we've included a `dagster.yaml` to configure settings for our instance and `pyproject.toml` to define our project as a module. This allows us to run [`dagster dev`](/guides/dagster/running-dagster-locally) without supplying command line arguments.

### Dagster Cloud

Depending on the type of deployment you're using in Dagster Cloud - Serverless or Hybrid - your project structure might look slightly different. Click the tabs for more info.

In cases where you want to configure [settings for a full deployment](/dagster-cloud/developing-testing/deployment-settings-reference), use the `deployment_settings.yaml` file.

#### Hybrid deployment

For Dagster Cloud Hybrid deployments, the `dagster_cloud.yaml` file is used to configure your code locations as well as other settings.

To customize your [Hybrid agent's configuration](/dagster-cloud/deployment/agents/customizing-configuration), include a `dagster.yaml` with the desired settings.

In cases where you want to configure [settings for a full deployment](/dagster-cloud/developing-testing/deployment-settings-reference), use the `deployment_settings.yaml` file.

For a Hybrid deployment to Dagster Cloud, an example project might look like this:
